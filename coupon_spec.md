# マタキテ 回数券機能 仕様書

> このファイルは回数券機能専用の仕様書です。既存の仕様書とは別ファイルで管理してください。

---

## 1. 概要

小規模サロン向けの「紙の回数券をデジタル化する」機能。  
決済は Stripe の支払いリンクを利用し、アプリ側は決済処理を一切持たない。  
購入後の回数券付与は店舗スタッフが手動で行うため、サポート負荷が極めて低い。

---

## 2. 基本思想

- 決済は Stripe に完全委任（Webhook・自動付与は行わない）
- 店舗は QR コードを提示し、顧客がスマホで支払うだけ
- 完了画面を見せてもらい、店舗が回数券を手動付与
- 紙の回数券と同じ運用フローで混乱がない
- サポートが増えない構造を最優先

---

## 3. 回数券購入フロー（全体）

1. 店舗が Stripe で「支払いリンク」を作成
2. マタキテ管理画面にリンクを登録
3. アプリがリンクから QR コードを自動生成
4. 店舗が QR コードを提示
5. 顧客がスマホで読み取り、Stripe で決済
6. 顧客が完了画面を店舗に提示
7. 店舗スタッフがマタキテで顧客QRコードを読み取り顧客を特定
8. 店舗スタッフが該当の回数券を手動付与
9. 顧客画面に「残り回数」が反映される

---

## 4. 認証・アカウント仕様

### 4.1 ログイン方法
- 顧客・店舗スタッフともに**メールアドレス＋パスワード**でログイン
- パスワードは暗号化して保存（平文での保存は禁止）

### 4.2 顧客アカウント登録
以下の**どちらの方法でも**登録できる：

- **顧客自身が登録**：アプリの登録画面からメールアドレスとパスワードを入力
- **店舗スタッフが代理登録**：管理画面の「代理登録」ボタンから顧客の情報を入力して登録

### 4.3 代理登録時のパスワード仕様

- スタッフが仮パスワードを設定して登録する
- 登録時に `password_changed = false`（パスワード未変更）フラグを付ける
- 顧客が初めてログインした際、自動でパスワード変更画面に遷移する
- 顧客がパスワードを変更すると `password_changed = true` に更新され、以降はホーム画面に直接進む

**顧客自身が登録した場合との違い：**

| 登録方法 | password_changedの初期値 | 初回ログイン後の動作 |
|----------|--------------------------|---------------------|
| 顧客自身が登録 | `true`（変更済み） | そのままホーム画面へ |
| スタッフが代理登録 | `false`（未変更） | パスワード変更画面へ自動遷移 |

### 4.3 顧客QRコード
- 顧客アカウント登録時に**その顧客専用のQRコード**を自動発行する
- QRコードには顧客IDが埋め込まれている
- 顧客はスマホのアプリ画面にQRコードを表示できる
- 店舗スタッフはこのQRコードを読み取ることで顧客を特定する

## 5. 画面仕様

### 5.1 店舗管理画面：回数券設定

| 項目 | 内容 |
|------|------|
| 回数券名 | 例：5回券、10回券 |
| 回数 | 例：5、10 |
| Stripe支払いリンク | URLを入力 |
| QRコード | リンクから自動生成・表示 |
| ダウンロード・印刷 | QRコード画像を保存・印刷できる |

### 5.2 店舗管理画面：回数券付与

| 入力項目 | 内容 |
|----------|------|
| 顧客特定 | 顧客QRコードを読み取って自動入力 |
| 回数券の種類 | 付与する回数券を選択（例：5回券） |
| 付与回数 | 例：5 |
| スタッフ名 | 手入力 |
| 理由 | 例：回数券購入 |

### 5.3 店舗管理画面：回数券消化

| 入力項目 | 内容 |
|----------|------|
| 顧客特定 | 顧客QRコードを読み取って自動入力 |
| 回数券の種類 | 消化する回数券を選択 |
| 消化回数 | 通常は1 |
| スタッフ名 | 手入力 |
| 理由 | 例：施術1回分 |

### 5.4 顧客画面

| 表示項目 | 内容 |
|----------|------|
| 店舗名・住所 | 大きく表示（不正防止） |
| 自分のQRコード | 大きく表示（スタッフに読み取ってもらう） |
| 回数券一覧 | 種類ごとに残り回数を表示 |
| 利用履歴 | 付与・消化の日時・スタッフ名・理由を表示 |

## 6. データ構造

### 6.0 users（ユーザー情報）
顧客・スタッフのアカウント情報を管理する。

| カラム名 | 内容 |
|----------|------|
| user_id | 一意のID（UUID） |
| email | メールアドレス |
| password | 暗号化されたパスワード |
| password_changed | パスワード変更済みか（true/false） |
| role | 役割（customer：顧客 / staff：スタッフ） |
| store_id | 所属店舗ID |
| created_at | 作成日時 |

> ⚠️ スタッフが代理登録した顧客は `password_changed = false` で登録する。顧客が初回ログイン時にパスワードを変更したら `true` に更新すること。

### 6.1 tickets（回数券マスタ）
店舗が設定する回数券の種類を管理する。

| カラム名 | 内容 |
|----------|------|
| ticket_id | 一意のID（UUID） |
| store_id | どの店舗の回数券か |
| name | 回数券名（例：5回券） |
| count | 回数（例：5） |
| stripe_link | StripeのURL |
| created_at | 作成日時 |

### 6.2 user_tickets（顧客の残回数）
顧客が持っている回数券の残数を種類ごとに管理する。

| カラム名 | 内容 |
|----------|------|
| user_ticket_id | 一意のID（UUID） |
| user_id | どの顧客か |
| store_id | どの店舗か |
| ticket_id | どの種類の回数券か（ticketsテーブルと紐づけ） |
| remaining_count | 残り回数 |
| created_at | 作成日時 |

> ⚠️ 同じ顧客が複数種類の回数券を持てる。ticket_idで種類を区別すること。

### 6.3 ticket_logs（回数券履歴）
付与・消化の操作履歴を残す。

| カラム名 | 内容 |
|----------|------|
| log_id | 一意のID（UUID） |
| user_id | 対象の顧客 |
| store_id | 対象の店舗 |
| ticket_id | 対象の回数券の種類 |
| change | 変化量（例：+5、-1） |
| staff_name | 操作したスタッフ名 |
| reason | 理由（例：回数券購入、施術1回分） |
| created_at | 操作日時 |

## 7. Stripe 連携仕様

### 7.1 使用するStripe機能
- 支払いリンク（Payment Link）のみ
- Stripe側で領収書メール送信
- Stripe側で決済完了画面表示

### 7.2 使用しない機能（禁止）
- Webhook
- Checkoutセッションの埋め込み
- 定期課金（サブスク）
- 自動付与

### 7.3 決済後の処理
- 顧客が完了画面を店舗に提示
- 店舗が手動で回数券を付与
- マタキテ側は決済情報を保持しない

---

## 8. 不正防止仕様

- 店舗名・住所を全画面に表示（店舗の混同を防ぐ）
- スタッフ名は毎回手入力（紙文化の再現・責任の明確化）
- 履歴に「誰が」「何の理由で」付与/消化したかを必ず記録
- 顧客データは store_id で完全分離（他店舗のデータは見えない）

---

## 9. サポート負荷を増やさないための制限

- 自動付与なし
- Webhookなし
- CSV出力なし
- グラフなし
- カスタマイズ項目を最小限に
- 決済トラブルは Stripe 側に完全委任

---

## 10. バリデーション（入力チェック）仕様

- 回数は1以上の整数のみ許可
- Stripe支払いリンクはURLの形式チェックを行う
- スタッフ名・理由は空欄不可
- 残り回数が0の場合は消化操作を禁止し、日本語でエラーを表示

---

## 11. 料金モデル

- マタキテ本体：月額2,000円（店舗単位・顧客無制限）
- 回数券機能：追加料金なし（オプション機能として提供）
